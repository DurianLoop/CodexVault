# Codex Vault v0.2 Upgrade Plan

This document defines the next milestone after the v0.1 internal beta.

v0.1 proves that Codex Vault can scan a local `.codex` folder, summarize important assets, safely export/import packs, simulate share-code migration, expose a small backend API lab, and make achievements interactive.

v0.2 should turn that local beta into a more trustworthy product-shaped build.

## Goals

1. Move from local-only simulation to a real sync-ready architecture.
2. Package the Windows app so testers do not need to install Python manually.
3. Improve import/export confidence with better previews, diffs, and rollback.
4. Make achievements feel more like a durable product feature.
5. Preserve the low-permission, low-surprise design of v0.1.

## Non-Goals

- Do not build a large social platform in v0.2.
- Do not expose public marketplace publishing before permission and moderation are designed.
- Do not require administrator privileges.
- Do not force users to upload private `.codex` content.
- Do not replace local-first workflows with cloud-only flows.

## Track 1: Real Account And Sync Foundation

### v0.2 Requirements

- Real account model with email or username login.
- Password hashing and server-side session/token storage.
- Device registry with device name, platform, first seen, last seen, and status.
- Pack ownership: every uploaded pack belongs to a user.
- Private share codes by default.
- Optional expiration for share codes.
- Download audit trail: who downloaded which pack and when.
- Server-side pack size limits.

### Suggested Implementation

- Keep the current Python backend as the reference API.
- Add a production-ready backend option later, such as FastAPI or another minimal web framework.
- Keep SQLite for local development.
- Add PostgreSQL-compatible schema design before public deployment.
- Store packs as files or object storage blobs, not directly inside a relational table for large payloads.

### Acceptance Criteria

- A user can register, login, bind a device, upload a pack, generate a private share code, and download it on another device.
- Invalid tokens cannot access private pack downloads.
- Expired share codes cannot be used.
- A user's pack list only shows their own packs.

## Track 2: Windows Packaging

### v0.2 Requirements

- Build a portable Windows package that does not require users to install Python manually.
- Keep a source-run mode for developers.
- Include a signed or clearly identified executable when signing is available.
- Keep all runtime data outside the install directory.

### Candidate Paths

1. **PyInstaller first**
   - Fastest path from the current Python/Tkinter app.
   - Good for v0.2 beta distribution.
   - Requires testing antivirus false positives and path handling.

2. **Tauri later**
   - Better long-term app shell.
   - Requires Rust, frontend build pipeline, and more packaging work.
   - Better candidate after backend and product flows stabilize.

3. **Electron later**
   - Mature desktop ecosystem.
   - Larger runtime footprint.
   - Not ideal for the current low-dependency beta goal.

### Acceptance Criteria

- A tester can download a zip, run the app, scan `.codex`, export a pack, import a pack, and view achievements without installing Python.
- The packaged app passes the same import/export tests as source mode.

## Track 3: Safer Import And Export UX

### v0.2 Requirements

- Export preview before creating a pack.
- Import preview before writing files.
- File-level diff for markdown assets.
- Conflict handling: skip, replace, rename, or merge manually.
- Rollback from the pre-import backup.
- Clear warning when a pack contains unknown folders.

### Acceptance Criteria

- The app can show what will be added, changed, and skipped before import.
- The user can cancel before any write.
- The user can restore from the backup created before import.

## Track 4: Achievement System Upgrade

### v0.2 Requirements

- Persist unlocked achievements across app launches.
- Store progress snapshots locally.
- Support hidden achievements and visible achievements separately.
- Support progress bars for count-based and time-based achievements.
- Add a small notification history.
- Add an asset pipeline for achievement icons.

### Design Direction

- Keep the current dark, game-platform feeling.
- Use playful binary/developer visual language.
- Do not make the achievement wall draggable.
- Keep implementation requirements visible only after clicking a card.

### Acceptance Criteria

- Unlock state survives app restart.
- Progress updates after scan/export/import actions.
- Clicking an achievement shows requirement and current progress.
- At least one notification appears when a test achievement is unlocked.

## Track 5: Product Documentation

### v0.2 Requirements

- Add a user guide with screenshots.
- Add a developer guide for running tests and backend.
- Add an API reference for backend endpoints.
- Add a security note explaining what is excluded from exports.
- Add a migration guide from v0.1 to v0.2.

### Acceptance Criteria

- A new tester can run the app from README alone.
- A contributor can run tests and understand the module layout.
- The security boundary is explicit and easy to audit.

## Track 6: Automated Validation

### v0.2 Test Matrix

- Unit tests for scanning, sensitive exclusion, backup, export, import, and manifest validation.
- Backend API tests for auth, device binding, upload, metadata, and download.
- UI smoke tests for importability and core state models.
- Pack roundtrip tests using a temporary `.codex` fixture.
- Real local `.codex` dry-run summary test that does not upload private content.

### Optional UI Automation

Tkinter UI automation is possible but fragile. For v0.2, prefer testing the app state models and backend workflows first, then add visual smoke checks only where stable.

## Suggested v0.2 Milestones

### Milestone A: Packaging Beta

- PyInstaller build works on Windows.
- Portable zip contains app, backend launcher, README, and tests.
- Source mode still works.

### Milestone B: Sync Beta

- Real backend deployment target selected.
- User-owned packs and private share codes implemented.
- Device-based download flow tested.

### Milestone C: Import Confidence

- Import preview and rollback are implemented.
- Markdown diff view is available.
- Unknown file warnings are visible.

### Milestone D: Achievement Polish

- Persistent achievement state.
- Notification history.
- Icon set draft.
- Progress events wired to scan/export/import flows.

## Recommended v0.2 Definition Of Done

- Windows packaged build is usable by a non-developer tester.
- No sensitive files are included in exported packs by default.
- Backend auth and share-code APIs are covered by automated tests.
- Import preview and rollback are covered by tests.
- Achievement unlock/progress state survives restart.
- README and user guide match the actual app behavior.
